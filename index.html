도입 배경

기존 Windows Hyper-V 기반 사내 장비의 성능 이슈와 확장성·유지보수 한계가 있었습니다. 서버 추가 구매를 검토했으나 메모리 가격 급등으로 기존 약 천만원 서버가 사천만원까지 비용이 증가하여, 하드웨어 대신 소프트웨어로 해결하기로 결정했습니다.

기술적 챌린지

  • 네트워크 설계: 특정 EMR은 인터넷 통신 차단, 특정 EMR은 브로드캐스트 허용 등 요구사항별 vLAN 설계가 가장 난이도 높았음
  • 스토리지 구성: Host는 LVM, Disk 이미지는 NAS 공유, 백업은 Backup Server → S3 저장 및 임포트 복제 구조
  • 네트워크 구성: Bridge, NAT, Host 등 사용 목적별 구성 + EMR 간 통신을 위한 다중 vLAN

의사결정

  • Proxmox 선택: VMware·OpenStack과 비교 후, 라이선스 비용 없이 확장성·유지보수가 가능한 Proxmox를 선택. KVM/LXC 지원으로 빠른 성능 제공, 클러스터 구조로 UI 통합 관리 가능

수행

  • 데스크탑 PC를 서버 케이스로 이관, Proxmox OS 셋업 및 클러스터 구성
  • 스토리지·네트워크·백업 전략·알림 체계(OTel + Slack) 구축
  • 사내 EMR 마이그레이션 및 부서별 EMR 환경 제공
  • 인증서 기반 사내 도메인 구성 — 검증된 사용자만 접근 가능한 관리 체계

인프라 아키텍처 고도화 및 비용 절감

2025 기술블로그 ↗
⌃
비용 20% 절감 ARM(Graviton) 이전 Karpenter 도입

도입 배경

한국 리전에서 c6a가 미지원되어 저성능 인스턴스로 운영 중이었습니다. 인프라 비용 절감과 성능 향상을 위해 Graviton(ARM)으로의 전환을 검토하고, Karpenter를 통해 서비스 목적에 맞는 다양한 인스턴스 풀을 On-demand+Spot 조합으로 구성하고자 했습니다.

기술적 챌린지

  • ARM 호환성: 개발자가 Mac(ARM)에서 로컬 빌드하여 호환성 문제는 적었으나, Self-hosted Action Runner의 골든 이미지를 ARM용으로 재구성하면서 패키지 버전 호환 문제를 해결했습니다. ARM을 지원하지 않거나 베타 단계인 서드파티는 x86 노드풀에 분리 배포했습니다.
  • 마이그레이션 순서: 서드파티 → 개발 → 스테이지 → 운영 순서로 진행. 멀티아키텍처 빌드(x86+ARM)로 빌드하여 노드풀 CPU Type 변경만으로 순차 전환하고, 인프라 리소스와 트레이스 로그 기반 모니터링으로 서비스 로직 검증.
  • Spot 장애 경험: Spot 비율 100% 설정 시 AWS Interruption으로 드레인 과정에서 Probe 연결 지연 발생. On-demand:Spot 비율을 조정하고, Affinity/TopologySpread 정책으로 On-demand 우선 배포, GracefulShutdown·Probe·PreStop 설정을 어플리케이션/인프라 양쪽에서 동시에 변경하여 해결.

의사결정

  • Karpenter vs Cluster Autoscaler: 기존 Autoscaler에서 서비스별·언어별 인스턴스 타입이 상이하여 비용 효율적 사용 불가. Karpenter Beta부터 도입 검토 — 빠른 노드 확장, 다양한 인스턴스 활용, NodePool 설계 기반 고성능/비용 효율 노드 확보.
  • SaaS→OSS 전환: 일부 기능만 사용하면서 비용이 크게 발생. OSS 전환 시 모니터링·로깅·트레이싱 연관 조회 개선, 비용 절약 규모, 고가용성/확장성 확보 방안을 사전 테스트로 검증하여 경영진 동의 확보.
  • On-demand:Spot = 1:9: Spot 100%에서 Interruption 장애 경험 후, 비업무 시간에도 병원 예약·검색 트래픽이 존재하므로 On-demand로 최소 안정성 확보 후 피크타임에 Spot을 최대 활용하는 비율로 결정.

수행

  • SaaS → OSS 단계 전환 (Grafana Stack+Thanos, Postman → Hoppscotch 등)
  • x86 → ARM(Graviton) 이전 + RI/SP 병행 적용
  • Karpenter 노드풀 설계: API(Spot 리밸런싱 허용) / Kafka Consumer(Spot 격리) / Argo Workflow(On-demand+Spot 분리)
  • GitHub Reusable Workflow로 CI 표준화

기여도

비용 누수 파악부터 월별 트래킹 리포트까지 전 과정 주도
  • 인프라 전체 비용 20% 절감 — SaaS→OSS 전환, ARM(Graviton) 이전, Karpenter Spot 전략 조합으로 달성

Amazon Q 및 AWS 보안 서비스를 활용한 위협 탐지 및 대응 자동화

2025 ⌃
보안 대응 70%+ 개선 API 요청량 20%+ 감소

도입 배경

보안 탐지·분석·보고가 수작업 중심이라 대응 리드타임이 길고 누락 위험이 존재. 암호화되지 않은 민감 정보도 존재했습니다.

기술적 챌린지

  • GuardDuty 튜닝: 오탐보다는 레거시 서비스의 보안 위반 사례를 대거 발견 — 퍼블릭 S3 엔드포인트 접근, 비허가 IP의 kubectl API 접근 등. 예외 처리가 필요한 케이스를 하나씩 분석·정리. GuardDuty 보고서 최대 90일 제한이므로 매월 데이터를 다운로드하여 별도 리포트로 관리.
  • WAF 오탐 대응: Block된 정상 트래픽에 대해 재시도 후 Block 사유 확인, 개발 환경에서는 Allow 시에도 로그를 남겨 오탐 발생 시 예외 처리 Rule/Group 구성. 대외 서비스는 고객사 IP 화이트리스트 + 정책 룰을 탑다운(우선순위) 방식으로 설계.
  • Secret 마이그레이션: GitHub 평문 Secret → Parameter Store 전환 시 네이밍 룰을 기획하고, ExternalSecret으로 Parameter Store 값을 불러와 사용. GitHub 평문 Secret 제거 및 권한 분리를 통한 암호화 저장 달성.

의사결정

  • Amazon Q 선택: 외부 서비스에 보안 데이터를 노출하지 않고 AWS 환경 내에서 처리하기 위해 선택. GuardDuty 내보내기 → S3 저장 → Amazon Q + Lambda 배치로 일간/월간 리포트 자동 생성.
  • 파이프라인 구조: GuardDuty → 원본 S3 저장 → Amazon Q Lambda → SNS 발송 + 일간/월간 S3 저장

수행

  • GuardDuty→Security Hub→Amazon Q 자동화 파이프라인
  • WAF 도입 및 오탐 방지 룰 튜닝
  • Trivy 이미지 스캔 + SonarQube 정적 분석 CI 통합
  • Helm Secret → Parameter Store(ExternalSecret) 전환

기여도

GuardDuty→Security Hub→Amazon Q 파이프라인 설계 및 WAF 룰 튜닝 담당
  • 보안 대응 속도 70%+ 개선 — 수동 탐지·분석 프로세스를 GuardDuty→Amazon Q 자동화 파이프라인으로 전환
  • API 비정상 요청 20%+ 감소 — WAF NoUserAgent·SQLi·SizeRestrictions 룰 튜닝으로 달성

SLI/SLO 체계 도입 및 서비스 안정성 개선

2025 ⌃
SLO 99.9% 3개월 달성 DB 성능 30% 향상 P90 <1s

도입 배경

가용성·응답시간 목표가 정량화되지 않아 장애 우선순위 판단과 배포 리스크 관리 기준이 부재했습니다.

기술적 챌린지

  • 비즈니스 메트릭 설계: 개발팀과 협의하여 접수·예약 완료율 등 핵심 플로우의 정상/비정상 지표를 Counter/Gauge 기반으로 수집. 주요 서비스(예약, 병원 조회) 간 장애 전파를 감지할 수 있는 메트릭 구성.
  • Slow Query 협업: Grafana에서 DB 리소스 메트릭에 Warning/Critical 기준 수립, Trace의 MongoDB 쿼리 Span 시간(배치 제외 1초 이상)에 알림 체계 구축. 각 담당자에게 분석 요청 → Index 추가 또는 병목 해소로 해결.
  • Error Budget: 소진 시 알림이 담당자에게 전달되어 즉시 분석 체계 가동.

의사결정

  • SLO 목표 수립 과정: 최초 도입 시 목표를 먼저 세우지 않고 메트릭을 수집. 당시 서비스 안정성 98% → 매월 99% → 99.5% → 99.9%로 단계적으로 목표를 높이며 병목 구간을 개선.
  • P90 선택 이유: P99·P95에서 발생하는 노이즈를 줄이기 위해 P90을 기준으로 선택.

수행

  • OpenTelemetry 트레이스 + Tempo 기반 비즈니스 메트릭 대시보드 구성
  • SLI(가용성·P90 응답시간) 정의 및 SLO 99.9% 수립
  • Error Budget 소진 알림 + 배포 제한 자동화
  • 정기 리뷰 미팅 기반 Slow Query·Query Targeting 개선

기여도

SLI 지표 정의, Grafana 대시보드 구축, 월간 정기 미팅 및 쿼리 최적화 협업 주도
  • SLO 99.9% 달성 — 도입 후 3개월 연속 달성 (98% → 99% → 99.5% → 99.9% 단계적 목표 상향)
  • P90 응답시간 <1s 달성 — 정기 리뷰 기반 Slow Query·Query Targeting 개선으로 DB 성능 30% 향상

기술적 챌린지

  • SaaS→OSS 전환: Datadog의 많은 기능을 사용하지 않으면서 비용이 크게 발생. 로그·트레이싱·인프라 리소스가 파편화되어 통합 옵저버빌리티를 구성하고자 전환 결정.
  • 데이터 복구: S3 생명주기 정책으로 원하는 시점의 데이터 복구가 가능해진 것도 OSS 전환의 장점.

의사결정

  • Loki 선택: 기존 OpenSearch에서 전환. 통합 옵저버빌리티를 위해 Grafana Stack으로 통일.
  • Alloy 선택: 기존 Fluent-bit Config를 참고하여 Alloy Config로 전환. Promtail은 추가 유지보수가 중단되어 Alloy 선택.
  • Tempo 선택: Loki와 동일한 사유 — Grafana Stack 통합 및 유지보수 지속성.

팀 도입

SaaS 종료 전 개발 환경을 먼저 OSS로 전환하여 대시보드·메트릭·알림 체계를 사전 정의. 운영 환경 이관 시 큰 문제 없이 전환 완료. 개발자 대시보드 구성은 DevOps에서 제공 (서비스 트레이스 상세, 로그 디버그, SLO 지표 등).

수행

  • Alloy 단일 수집 에이전트: Loki(로그)·Tempo(트레이스)·Thanos Remote Write(메트릭)
  • Prometheus + Thanos 멀티 클러스터 메트릭 통합
  • Grafana Correlation 대시보드 구성
  • K8s·Kafka·DB·어플리케이션 통합 대시보드 20+ 및 알림 체계
