MVP 개발 바이브코딩 투표시스템

도입 배경

사내 간식 주문을 수작업으로 요청·처리하고 있어 매주 반복되는 운영 비효율이 누적되었고, 여러 직원의 간식 도입 요청을 취합할 방법이 없었습니다. AI 도구를 활용한 빠른 자동화로 반복 업무를 줄이고 복지 운영을 개선할 기회로 판단했습니다.

기술적 챌린지

  • 투표 동시성 충돌 가능성: 다수 사용자가 같은 시점에 투표할 경우 집계 정합성이 깨질 수 있음. 추가 인프라 없이도 동시성을 안전하게 제어할 방안 필요.
  • 비개발자 유지보수 제약: 담당자 교체 이후에도 운영 지속 필요했으나, 개발자가 아닌 운영자도 데이터를 직접 확인·수정 가능해야 함. 일반 DB는 학습 비용·접근 권한이 진입장벽.
  • MVP 속도 ↔ 견고성의 trade-off: 단기간 내 배포 가능한 단순함과 이후에도 재사용 가능한 구조를 동시 만족 필요. 과한 설계는 일정 초과, 과소 설계는 유지보수 불가.

의사결정

  • 바이브코딩 + Claude 기반 MVP: 프레임워크 선택·DB 설계부터 시작하는 전통적 접근은 일정상 불가능. Claude 프롬프트(Skill·Hook) 기반 반복 생성·검증으로 빠르게 동작 가능한 수준까지 도달.
  • Excel 데이터 저장 (vs RDB): 사용자·데이터 규모가 작고 비개발자 운영자가 직접 다뤄야 하는 요구사항 고려. 트랜잭션 성능보다 접근성이 중요한 맥락이라 Excel 채택.
  • 인메모리 락 (vs 분산 락·DB 트랜잭션): 단일 인스턴스 내 동시성만 다루면 되는 규모라 Redis/DB 락은 과잉. 인메모리 락으로 구현 단순화 + 충돌 회피.
  • 상품 정보 크롤링 자동화: 운영자가 매 회차 수기 입력하는 부담을 제거하기 위해 외부 상품 페이지 자동 수집 경로 설계.

수행

  • Claude 기반 프롬프트 아키텍처(Skill·Hook) 설계 및 바이브코딩으로 백엔드·프론트엔드 일괄 구성하여 MVP 신속 완성.
  • 인메모리 락으로 투표 요청을 직렬화하여 동시 요청 상황에서도 집계 정합성 확보.
  • Excel 기반 데이터베이스 구조 설계로 운영자가 시트 편집만으로 회차·상품·한도 직접 관리 가능.
  • 매주 회차 자동 생성과 주문 집계 파이프라인 구성으로 운영 부담 최소화.
  • 외부 상품 페이지 크롤링으로 상품명·이미지 자동 수집하여 등록 과정 간소화.

결과

  • 바이브코딩으로 6시간 만에 MVP 구성 완료 — 기획부터 배포까지
  • 투표 기반 간식 구매 자동화로 수작업 요청·처리 업무 시간 절감
  • 비개발자 운영자가 직접 회차·상품 관리 가능한 유지보수 구조 정착

서비스 시연 영상

하이퍼바이저 → KVM 전환 가용성 확대 성능 및 비용 개선

도입 배경

기존 Windows Hyper-V 기반 사내 가상화 장비의 성능 한계와 확장성·유지보수 부담이 누적되었고, 메모리 시세 급등으로 서버 증설 단가가 수 배 상승해 하드웨어 증설만으로 해결하기 어려운 상황이었습니다. 기존 자원을 재활용하면서도 확장성·관리성을 동시에 확보할 수 있는 소프트웨어 기반 대안이 필요했습니다.

기술적 챌린지

  • 혼재된 네트워크 요구사항: 사내 여러 레거시 서비스가 서로 다른 통신 정책(인터넷 차단·브로드캐스트 허용·격리 통신 등)을 요구해, 단일 플랫폼 위에서 격리와 상호 통신을 동시에 만족시킬 네트워크 설계 필요.
  • 스토리지 계층의 역할 분리: 런타임 성능·공유 접근성·백업 복구 속도 동시 고려 필요, 단일 스토리지로 묶으면 한 영역 장애가 전체로 번질 위험 존재.
  • 단일 관리 콘솔에서의 다중 물리장비 통합 운영: 여러 물리 서버를 개별 접속·관리할 경우 운영 부담이 증가하므로, 하나의 관리 콘솔에서 다수 노드를 클러스터로 묶어 가상머신·스토리지·네트워크를 일괄 제어할 수 있는 플랫폼 필요.

의사결정

  • Proxmox VE 채택 (vs VMware, OpenStack): 라이선스 비용 없이 KVM·LXC·클러스터·UI 통합 관리를 제공해 사내 규모에 적합. VMware는 비용·락인, OpenStack은 운영 복잡도가 과잉이라 판단.
  • 기존 하드웨어 재활용 (vs 신규 서버 구매): 메모리 시세 급등기에 신규 구매는 비용 효율이 크게 악화. 기존 데스크탑 PC를 서버 케이스로 이관해 초기 투자 최소화.
  • 스토리지 다층 구성 (LVM + NAS + Backup Server → S3): Host 런타임은 LVM으로 성능 확보, VM 디스크는 NAS 공유로 이동·복제 용이, 백업은 Backup Server 경유 S3 복제로 DR까지 포함. 단일 스토리지로 묶지 않아 장애 격리.
  • Bridge·NAT·vLAN 혼합 네트워크: 서비스별 요구에 맞춰 네트워크 모드 분리, vLAN으로 통신 정책 차등 적용. 단일 네트워크 통합 대안은 보안·장애 전파 리스크로 배제.

수행

  • 기존 데스크탑 PC를 서버 케이스로 이관 후 Proxmox VE 설치 및 클러스터 구성 완료, 노드 간 HA 기반 관리 체계 확보.
  • Bridge·NAT·Host 모드 혼합과 다중 vLAN 구성으로 서비스별 통신 정책(격리·브로드캐스트 허용·인터넷 차단 등) 개별 적용.
  • Host는 LVM, VM 디스크는 NAS 공유, 백업은 Backup Server → AWS S3 복제로 이어지는 다층 스토리지 구조 설계·구축.
  • 정기 스냅샷과 S3 복제 결합으로 클러스터 간 빠른 복제 체계 구축, 장애 시 VM 단위 복원 가능 구성.
  • OpenTelemetry 기반 메트릭 수집과 Grafana 대시보드·Slack 알림 연동으로 장애·리소스 임계치 조기 탐지.
  • 사내 인증서 기반 도메인 구성으로 검증된 사용자만 Proxmox UI 접근 가능하도록 제한.

결과

  • 추가 서버 구매 없이 기존 대비 약 40% 이상 VM 증설 가능 — 기존 하드웨어 재활용 기준
  • VM 복제 기반 신규 서버 세팅 시간 약 1분으로 단축 — 기존 수동 OS 설치 대비
  • AWS S3 백업 전략으로 클러스터 간 빠른 복제·DR 경로 확보
  • 약 50개 VM 통합 운영 중 — 부서별 독립 워크로드 환경 제공

Grafana 기반 모니터링 (OTel) 예시

ARM(Graviton) 적용 Karpenter 도입 비용 20% 절감 Spot 도입

도입 배경

서비스 확장에 따른 EKS 노드 수요 증가로 월 인프라 비용이 지속 상승하는 상황이었고, 리전 제약으로 이전 세대 인스턴스 운영이 강제되어 가격 대비 성능 열세가 누적되었습니다. SaaS 기반 Observability는 사용량 대비 고정 비용 비중이 커 비용 구조 전반의 재설계가 필요했습니다.

기술적 챌린지

  • Spot 가용성 ↔ 비용의 균형점: 상시 서비스 트래픽이 존재해 100% On-demand는 비용 비효율, 100% Spot은 AWS Interruption 시 drain 과정에서 Probe 연결 지연 리스크 존재. 두 극단 사이 허용 가능한 가용성 기준 설정이 핵심 고민.
  • ARM 전환의 경계선: ARM 빌드 시 일부 라이브러리·서드파티 바이너리의 ARM 지원 여부 검증 필요, 빌드 파이프라인(멀티아키 빌드·캐시·이미지 태깅)의 정상 동작 검증 요구. "어디까지 ARM으로, 어디는 x86으로 남길 것인가" 판단 필요.
  • 운영 리스크 없는 전환 경로: 멀티테넌트 EKS에 여러 서비스가 혼재된 환경에서 노드풀 CPU Type 전환의 단계화·롤백 가능성·회귀 검증 기준(메트릭·트레이스) 설계 필요.

의사결정

  • Karpenter 채택 (vs Cluster Autoscaler): 기존 CA는 노드 그룹 단위 확장으로 서비스별 인스턴스 다양성 수용 불가. Beta부터 PoC로 NodePool 기반 혼합 인스턴스·빠른 스케일 아웃·Spot 중단 선제 감지를 검증해 채택.
  • SaaS → OSS 전환: Grafana Stack+Thanos/Loki/Tempo PoC로 연관 조회·장기 보존·쿼리 성능 검증. 절감 규모·전환 리스크·운영 부담을 수치화해 내부 승인 확보.
  • On-demand : Spot = 1 : 9: Interruption 장애 학습을 반영, 상시 트래픽 구간의 최소 안정성은 On-demand로 확보하고 피크에는 Spot을 최대 활용. NodePool 별 Spot rebalancing 허용/불허 분리.
  • ARM 혼합 전략: 제어 가능 영역(앱·CI Runner·주요 노드풀)은 ARM, ARM 미지원·베타 서드파티는 x86 전용 노드풀로 격리.

수행

  • Affinity/TopologySpread로 On-demand 우선 배포 유도, GracefulShutdown·PreStop·Probe를 앱/인프라 양단 동시 적용, Karpenter Spot 중단 이벤트 기반 선제적 drain으로 Spot 환경 안정화.
  • 멀티아키텍처 이미지 빌드(x86+ARM) 후 노드풀 CPU Type 변경으로 무중단 순차 전환, Self-hosted Runner 골든 이미지도 ARM으로 재구성.
  • Karpenter NodePool을 3종(API · Kafka Consumer · Argo Workflow)으로 분리하여 워크로드별 스케줄링 정책 차등 적용, GitHub Reusable Workflow로 CI 파이프라인 표준화.

  • Karpenter Spot에 RI/Savings Plan 병행 적용으로 장기 보장 가격과 스팟 가변 가격을 혼합한 비용 구조 다각화.

결과

  • 인프라 전체 월 비용 20% 절감 — 도입 전후 기준, SaaS→OSS · ARM · Spot 조합 효과

보안 대응 속도 개선 크롤링/SQL 공격 요청량 감소 암호화

도입 배경

보안 탐지·분석·보고가 수작업 중심이라 대응 리드타임이 길고 누락 위험이 존재했습니다. 동시에 코드 및 CI 환경변수에 암호화되지 않은 민감정보가 산재해 있어, 탐지 자동화와 민감정보 관리 체계를 함께 정비할 필요가 있었습니다.

기술적 챌린지

  • 탐지 신호 노이즈 vs 실제 위협 구분: GuardDuty 활성화 초기, 레거시 서비스의 오래된 정책(퍼블릭 S3 접근·비허가 IP의 API 접근 등)이 대규모 탐지로 이어져 실제 위협과 예외 처리 대상을 구분할 기준 부재. 또한 GuardDuty 보고서 보존 기간(최대 90일)이 짧아 장기 추세 분석 어려움.
  • WAF 규칙 정밀도 — 오탐 vs 과소 탐지의 균형: 규칙이 너무 엄격하면 정상 트래픽도 Block되어 서비스 장애로 이어질 수 있고, 너무 느슨하면 공격 탐지 효과 부재. 서비스 유형별 트래픽 특성을 반영한 규칙 적용 순서와 예외 범위를 사전 확정 어려움.
  • 보안 데이터의 외부 노출 리스크: 탐지 이벤트 자동 분석에 외부 AI/LLM을 활용하면 보안 데이터가 서비스 경계 밖으로 유출될 가능성 존재. AWS 환경 내 폐쇄 파이프라인 구성 필요.

의사결정

  • Amazon Q 채택 (vs 외부 LLM/GPT API): 보안 이벤트 데이터를 외부 서비스에 노출하지 않고 AWS 환경 내에서 분석·리포트 자동 생성. 외부 LLM API는 보안 데이터 외부 전송 리스크로 배제.
  • GuardDuty + Security Hub 통합 파이프라인: GuardDuty 단독으로는 보존 기간 제한·크로스 서비스 연동 한계 존재. Security Hub를 통합 허브로 두고 S3 장기 저장 + Amazon Q Agent 자동 분석 구조로 설계.
  • WAF 단계적 적용 (Count → Block): 초기 Block 즉시 적용은 정상 트래픽 차단 리스크가 커, Count 모드로 탐지 로그 수집 후 오탐 범위 확인·예외 룰 구성 완료 시 Block 전환하는 방식 채택.
  • Secret 체계 전환 (GitHub 평문 → Parameter Store + ExternalSecret): 코드·CI 환경변수에 평문 노출된 민감정보를 Parameter Store로 이관. 네이밍 룰 설계 후 ExternalSecret으로 K8s 연동, 로컬 개발환경은 Infisical로 분리.

수행

  • GuardDuty → Security Hub 통합으로 탐지 이벤트를 S3에 장기 저장, Amazon Q Agent 배치 자동 분석으로 SNS 발송 및 일간·월간 리포트 생성 구성.
  • 서비스/목적별 WAF 구성 이후 정상·이상 트래픽 분류를 통해 NoUserAgent·SQLi·SizeRestrictions 룰을 Block으로 전환, 서비스 유형별 IP 허용목록·우선순위 규칙 설계.
  • 탐지 신호 분석으로 예외 처리 대상 분리, 90일 보존 한계 보완을 위한 월별 S3 아카이빙 체계 구축.
  • GitHub 평문 Secret을 AWS Parameter Store로 전환, 네이밍 룰 설계와 ExternalSecret K8s 연동 적용, Infisical 도입으로 로컬 환경변수 저장 차단.
  • Trivy 컨테이너 이미지 스캔과 SonarQube 정적 분석을 CI 파이프라인에 통합.

결과

  • 보안 대응 속도 70%+ 개선 — 대응 리드타임 단축
  • 크롤링·SQL 등 비정상 API 요청 20%+ 감소
  • 레거시 S3 퍼블릭 액세스 노출 탐지 및 조치, 내부 도메인 교체로 공격 표면 축소
  • 코드·CI 환경 전반 민감정보 암호화 저장 체계 전환 완료

SLO 목표 3개월 달성 서비스 품질 향상 레이턴시/슬로우쿼리 감소

도입 배경

가용성·응답시간 목표가 정량화되지 않아 장애 우선순위 판단과 배포 리스크 관리 기준이 부재했습니다. 경험과 직관에 의존하던 대응 방식에서 벗어나, 데이터 기반으로 서비스 품질을 측정하고 운영 리스크를 통제할 수 있는 체계가 필요했습니다.

기술적 챌린지

  • 비즈니스 메트릭 정의의 어려움: 인프라 메트릭(CPU·메모리)은 실제 서비스 품질과 직결되지 않아 비즈니스 레이어의 SLI 후보 선별 필요. 핵심 플로우별 정상/비정상 상태 기준에 대한 개발팀 합의 필요, 앱 코드 계측이 필요해 협업 의존도 높음.
  • SLO 목표치 — 너무 높으면 Error Budget 조기 소진, 너무 낮으면 형식화: 현황 파악 없이 목표를 먼저 설정하면 현실과 괴리 발생, 달성하기 쉬운 목표는 개선 동기 상실. 단계적 상향 방식이 적합하지만 각 구간에서의 병목 분리 기준 부재.
  • Error Budget 소진 시 배포 제한 정책 도입의 팀 내 저항: Error Budget 소진 시 배포를 제한하는 정책은 개발 일정과 충돌 가능성 존재. 단순 알림 이상의 조율 체계와 팀 내 공감대 형성 필요.

의사결정

  • SLO 단계적 목표 상향: 최초 도입 시 메트릭 수집을 먼저 하고 현황 파악 후 목표 설정. 98% → 99% → 99.5% → 99.9% 단계적 상향으로 각 구간에서 병목 발굴·개선.
  • P90 기준 응답시간 SLI (vs P99/P95): P99·P95는 극단 케이스 노이즈가 커 실질적 사용자 경험 반영이 어려움. P90이 서비스 다수 사용자의 응답 경험을 대표하는 지표로 판단.
  • 비즈니스 메트릭 계측: 인프라 메트릭만으로는 서비스 품질의 직접 지표가 되지 않아, OpenTelemetry 기반으로 핵심 비즈니스 플로우의 성공률·응답시간을 Counter/Gauge로 별도 수집.
  • Slow Query 임계값 1s 기준 (배치 제외): 배치 잡의 긴 쿼리를 포함하면 서비스 레이턴시 알림에 노이즈 과다. 배치 제외 1s 기준으로 분리해 실제 서비스 응답 병목만 탐지.

수행

  • 개발팀과 협업으로 핵심 비즈니스 플로우의 Counter/Gauge 메트릭 정의, OpenTelemetry + Tempo 트레이스 기반 계측 구성.
  • 가용성·P90 응답시간 SLI 정의, 단계적 SLO 목표 수립 및 Grafana 대시보드 구축.
  • Error Budget 소진률에 따라 Slack 채널 → 긴급도 기반 개인 멘션으로 이어지는 다단계 알림 체계 구성, 배포 제한 정책 자동화.
  • MongoDB 쿼리 Span 1s 기준 알림 구축, 정기 리뷰 미팅을 통해 담당자별 Index 추가·병목 해소 협업 주도.

결과

  • SLO 99.9% 달성 — 도입 후 3개월 연속 달성 (98% → 99% → 99.5% → 99.9% 단계적 목표 상향)
  • P90 응답시간 <1s 달성 — 정기 리뷰 기반 Slow Query·Query Targeting 개선으로 DB 성능 30% 향상
  • SLI 기반 이상 탐지 체계 확립으로 서비스 장애 발생 빈도 감소

빌드 3–4분 → 30초~1분 개발 생산성 향상 CI 표준화

도입 배경

Self-hosted Runner 도입 이후 Docker 빌드에 3–4분이 소요되어 반복 배포·핫픽스 시 개발 생산성에 부담이 누적되고 있었습니다. Runner가 Stateless로 동작해 레이어 캐시가 보존되지 않았고, 레포별 Dockerfile 구조도 제각각이라 캐시 전략과 빌드 표준화를 함께 정비할 필요가 있었습니다.

기술적 챌린지

  • Self-hosted Runner의 캐시 공유: Runner Pod는 실행 단위로 생성·소멸되어 로컬 Docker 레이어 캐시 재사용 불가. 기존 registry 기반 inline 캐시 방식은 코드 수정 없는 재빌드에서도 레이어 재사용이 제대로 되지 않아 캐시 효과가 제한적.
  • 공통 템플릿 부재: 레포마다 Base Image·Layer 순서·패키지 설치 단계가 달라 캐시 최적화를 일괄 적용 불가.
  • 동시성 이슈: 여러 Runner가 동일 캐시 저장소를 사용할 때 동시성 이슈가 발생하여 정상적인 빌드가 불가능함.

의사결정

  • ECR Cache 사용: ECR에 저장된 캐시 레이어를 가져와 재사용하고, 빌드 후 캐시를 ECR에 저장하는 구조로 변경.
  • ECR 생명주기 적용: ECR에 저장된 캐시 레이어를 무한히 저장하면 비용이 발생하여 생명주기 정책 적용.
  • 자체 노드 모듈 저장소 사용: 노드 모듈 캐시를 tar 파일로 저장하여 Self-hosted Runner에서 공유해 key 기반으로 사용하고, 새로운 빌드 시 tar 파일로 노드 모듈을 저장하여 재사용.

수행

  • Golden Image에 BuildKit·공통 빌드 패키지 구성을 통해 Runner 구동 시 이미지 Pull 비용 제거, Registry Rate Limit 회피.
  • Dockerfile 공통 템플릿(Layer 순서·의존성 설치 분리·Multi-stage 정리) 정립 및 서비스 레포 순차 이관, 캐시 히트율 최적화.
  • EFS를 통해 노드 모듈 패키지를 공유하여 노드 버전에 따른 모듈을 각각 구성하여 사용하고, 도커 템플릿을 통해 캐시 히트 최적화를 구성.

결과

  • Docker 빌드 속도 3–4분 → 30초~1분
  • 반복 배포·핫픽스 대응 시간 단축으로 개발 생산성 및 배포 빈도 향상
  • Dockerhub 장애와 무관하게 자체 빌드 체계로 빌드 안정성 확보
리소스 형상관리 모듈화 배포 프로세스 구성

도입 배경

콘솔 수작업 중심 운영으로 변경 이력과 배포 관리가 이루어지지 않았고, 환경 간 인프라 형상 불일치가 반복 발생했습니다. 변경 사이드이펙트를 예측하기 어려운 구조였고, 협업 시 누가 무엇을 변경했는지 추적할 수 없었습니다.

기술적 챌린지

  • 레거시 리소스 Import 범위·순서 결정: 수작업으로 생성된 기존 리소스를 어느 단위까지, 어떤 순서로 코드화할지 기준 부재. 사용 중/미사용 리소스 판별 어려움, EKS처럼 배포 목적 리소스와 서비스가 직접 참조하는 리소스 간 모듈 경계를 사전 정의 필요.
  • ArgoCD와의 리소스 소유권 충돌: ArgoCD가 자동 생성·관리하는 리소스와 Terraform 관리 리소스의 경계가 모호해 충돌 가능성 상존. 실제로 Terraform 정리 과정에서 ArgoCD가 참조 중인 리소스가 의도치 않게 제거되는 문제 발생.
  • State 백엔드 전환 — 이관 신뢰성과 Lock 정합성: TFC → GitHub + S3 전환 시 기존 State 무중단 이관, S3 Lock 신뢰성 확인, OIDC 인증 전환까지 동시 검토 필요.

의사결정

  • TFC 도입 (vs GitHub Actions + S3/DynamoDB): 초기 도입 시 TFC Free Tier가 현 규모에 충분하고, GitHub Actions + S3/DynamoDB 구성에서 State 데드락 이슈가 있었음. TFC의 UI 기반 Plan/Apply 제어로 가시성 확보.
  • TFC → GitHub + S3 전환 (가격 정책 변경 대응): TFC 가격 정책 변경 후 비용 효율 재검토. S3 네이티브 Lock 지원으로 DynamoDB 없이 State Lock 구성이 가능해져 전환 시점으로 판단. 기존 State 이관 + AWS AccessKey → OIDC 인증 전환으로 보안도 동시 강화.
  • Import 단계화 전략: 일괄 Import는 의존 관계·사이드이펙트 리스크가 높아 서비스 영향 없는 리소스부터 단계적으로 코드화. ArgoCD 충돌 이후 두 도구 간 소유권 경계를 문서화해 재발 방지.
  • 환경별 Workspace 분리: dev·stage·prod·mgmt 환경별 Workspace를 분리해 Plan/Apply 영향 범위를 격리, 공유 모듈 재사용 기반으로 샌드박스 자동화까지 확장.

수행

Grafana 올인원 구성 Observability 비용 절감 오픈소스 활용

도입 배경

모니터링·로깅·트레이싱 도구가 분산되어 장애 원인 추적이 지연되었습니다. 각기 다른 SaaS를 사용하면서 연관 분석이 어렵고 비용도 지속 증가해, 단일 관제 체계와 비용 구조 개선이 동시에 필요했습니다.

기술적 챌린지

  • 멀티 클러스터 메트릭 통합 — 장기 보존·전역 쿼리·카디널리티 제어의 동시 요구: Prometheus 단독 구성으로는 복수 EKS 클러스터의 장기 보존과 전역 쿼리 불가. 단일 Grafana에서 조회 가능하게 통합하면서도, 레이블 폭증으로 인한 비용·성능 저하를 함께 제어 필요.
  • 트레이스 보존 vs 비용의 trade-off: 전체 트레이스 보존 시 Tempo S3 비용 급증, 과도한 샘플링 축소는 에러·지연 재현에 필요한 트레이스를 놓칠 위험 존재. 디버깅 가치를 유지하면서 비용을 통제할 샘플링 전략 필요.
  • Loki 레이블 카디널리티 설계: Loki의 인덱스·청크 구조상 레이블 카디널리티가 비용과 쿼리 성능에 직결. 기존 로그 구조를 그대로 가져오면 고카디널리티 필드가 성능·비용 이슈를 유발할 위험 존재.

의사결정

  • SaaS → Grafana Stack OSS 전환 (vs SaaS): SaaS는 데이터·호스트 수에 비례해 비용이 선형 증가하는 반면 실제 활용 기능은 일부에 그치는 구조. OSS 스택은 필요한 지표·로그만 선별 수집하고 보존 정책을 직접 설계할 수 있으며, 단일 Grafana에서 메트릭·로그·트레이스 연관 탐색이 가능해 전환 결정.
  • Thanos 채택 (vs Prometheus 연합): 멀티 클러스터 장기 보존·전역 쿼리·Downsampling·hot/cold 계층 보존 정책을 Sidecar·Querier·Store·Compactor 구성으로 해결. Prometheus 연합은 장기 보존 한계 존재.
  • Alloy 채택 (vs Promtail·Fluent-bit): Promtail은 유지보수 중단 예정이고 Fluent-bit 대비 Loki와의 통합성·설정 일관성이 우수. 로그 수집 에이전트를 Alloy로 단일화하여 운영 부담 감소, 기존 Fluent-bit 구성 참조해 전환.
  • Tail-based 트레이스 샘플링: 에러·지연 임계 초과 트레이스는 전량 보존, 정상 트래픽은 비율 샘플링으로 비용·가시성 동시 확보. Head-based 샘플링은 에러 트레이스 누락 리스크로 배제.

수행

  • Alloy + Otel 에이전트로 로그·트레이스·메트릭 통합 수집, Prometheus + Thanos 기반 멀티 클러스터 통합과 S3 장기 보존·Downsampling 설계.
  • Loki 레이블 스키마 재설계 및 고카디널리티 필드의 구조화 로그 전환, S3 라이프사이클(Standard → IA → Glacier) 보존 정책 수립.
  • K8s·Kafka·DB·앱 통합 대시보드 20개 이상 구축, Grafana Correlation으로 메트릭·로그·트레이스 연관 탐색·알림 체계 구성.
  • SaaS 종료 전 개발 환경에서 OSS 스택을 선행 구축한 뒤 운영 환경 무중단 이관 완료, 개발자용 공용 대시보드(트레이스 상세·로그 디버그·SLO 지표) 제공.

결과

  • Observability 비용 70% 절감 — Datadog 등 SaaS에서 Prometheus·Thanos·Grafana Stack OSS로 전환
  • Grafana 기반의 올인원 대시보드 구성 — K8s·Kafka·DB·애플리케이션 메트릭-로그-트레이스 단일 관제 체계 달성

도입 배경

In-place 업그레이드 중 Istio 호환성 이슈로 서비스가 사용 불가한 상황이 발생해 복구 지연과 중단 위험이 상존했습니다. 롤백이 즉시 가능한 Multi-Target 방식으로 전환할 필요가 있었습니다.

기술적 챌린지

  • DNS TTL — 즉시 트래픽 전환 불가: 초기 Route53 레코드 변경 방식은 클라이언트 DNS 캐시 잔존으로 즉시 전환 불가. CloudFront 경로는 캐시 무효화로 대응 가능하지만, ELB 직결 경로는 TTL 영향이 남아 트래픽 전환 시점 정밀 제어 어려움.
  • Blue-Green 이중 운영 비용: 신·구 클러스터를 동시 운영하는 기간에 노드 비용이 두 배로 증가, 검증 기간을 길게 가져갈수록 비용 부담 선형 증가.
  • 상태 있는 워크로드의 안전한 전환 순서: Kafka 컨슈머 리밸런싱과 인서비스 배치 잡 중복 실행이 데이터 정합성 문제를 유발할 가능성 존재. 전환 순서와 상태 보장 기준 필요.

의사결정

  • In-place → Blue-Green(Multi-Target) 전환: In-place는 롤백 불가·호환성 검증 후 재시도 불가. 신·구 클러스터 병행 유지로 이상 감지 즉시 가중치 0%로 롤백 가능한 구조 채택.
  • Route53 → ALB 타깃그룹 가중치 전환: DNS TTL 캐시 영향 제거를 위해 ALB 타깃그룹 가중치 기반 10% → 50% → 100% 점진 전환으로 고도화. 즉시 트래픽 스위칭 가능.
  • 테스트 구간 Spot·프로덕션 전환 후 재프로비저닝: 이중 운영 비용 최소화를 위해 검증 구간은 Spot 인스턴스로 구성, 실제 트래픽 전환 시점에만 프로덕션 노드풀로 재프로비저닝.
  • K8s API 호환성·add-on·3rd-party 사전 검증 후 순차 적용: 업그레이드마다 API 제거·변경 및 Chart 호환성 사전 검증을 거쳐 순차 적용, 비호환 발견 시 전환 중단·롤백 기준 명확화.

수행

  • Multi-Target 클러스터 병행 운영과 함께 ALB 타깃그룹 가중치를 10% → 50% → 100%로 점진 전환, 이상 감지 시 0%로 즉시 롤백 가능하도록 구성.
  • 순차 전환 스크립트 구성으로 Kafka 컨슈머 리밸런싱·배치 중복 실행 방지.
  • 누락 API 검증용 프록시 서버 운영으로 구 클러스터 잔여 트래픽 추적 및 신규 클러스터로 라우팅.
  • 안정화 이후 3일 유예를 두고 구 클러스터 제거, Terraform IRSA 모듈로 멀티 클러스터 권한 일관성 확보.
  • 매니지먼트 계정 ArgoCD 기반으로 멀티 클러스터 배포 일관성 유지.

결과

  • 업데이트 소요 시간 10h → 5min으로 단축 — Multi-Target 가중치 전환 방식으로 달성

  • Gateway 이원화: 내부 통신은 성능 중심 신뢰 기반, 외부 진입점은 RateLimit·JWT·mTLS 기반 보안 강화로 역할 분리.
  • SAGA 패턴 채택: 분산 환경에서 잠금 점유로 확장성을 저해하는 기존 방식 대비, 보상 트랜잭션(compensating step)으로 트랜잭션을 분해해 점유 시간을 최소화.

수행

  • Jenkins 기반 빌드 프로세스를 GitLab 태그 기반 파이프라인으로 전환.

오픈소스 기반 인프라 구성 예시

Rancher 도입 K8s·DB 통합 모니터링 업데이트 속도 최대 45% 향상

도입 배경

ESL 레거시 시스템의 서비스 확장 요구와 MSA 전환을 위해, K8s 기반 오케스트레이션 배포·운영 체계가 필요했습니다. 기존 컨테이너 환경으로는 업데이트 처리 속도·트래픽 확장에 한계가 명확해 전면 재설계가 요구되었습니다.

기술적 챌린지

  • 대용량 업데이트 처리: 레거시 직렬 처리 구조에서 대용량 업데이트 처리 시간 과도. K8s 전환만으로는 처리 속도 개선 어려움, Pod/Thread 비율과 메시지 분산 처리의 최적 구성 기준 부재.
  • DB IO 병목: 대용량 테이블의 IO Wait 급증으로 DB 부하 상승, 누적된 dead tuple과 스토리지 한계 간 원인 구분이 선행 과제.
  • MQ HA: 노드 장애 시 일부 Queue 메시지 누락 확인, 클러스터 HA만으로는 메시지 유실 완전 방지 어려움.

의사결정

  • Rancher 도입: Kubernetes를 손쉽게 배포·운영할 수 있는 오케스트레이션 도구가 필요하여 Rancher 채택, 클러스터 생성·업그레이드·운영 일관성 확보.
  • Pod·Thread·MQ 경로 튜닝: 직렬 처리 구조의 병목 해소를 위해 Pod 수·Thread Factor·MQ 소비 경로를 실험적으로 조정하여 분산 처리 최적 구성 확보.
  • Vacuum 설정 튜닝 + SSD 전환 병행 검증: IO Wait 원인을 먼저 분리 검증하여 vacuum 누적 해소로 상당 부분 개선한 뒤 SSD 전환의 추가 효과를 순차 검증, 교체 선행 방식은 배제.
  • 재전송 + 중복 방지 처리 병행: MQ 이중화만으로는 유실된 메시지를 완전히 복구할 수 없어, 애플리케이션이 실패한 메시지를 다시 보내고 같은 메시지가 두 번 처리되지 않도록 검증하는 로직 추가.

수행

  • Rancher 기반 K8s 인프라 구축 및 코드 스캔·이미지 저장소·자동 배포를 포함한 GitOps 파이프라인 설계.
  • API Gateway·인증, 메시지 큐, 이중화 DB 등 핵심 공통 컴포넌트의 HA 아키텍처 설계·검증.
  • 통합 모니터링·로그·트레이싱 스택 구성, 애플리케이션 Probe·Graceful Shutdown 검증 주도 및 운영·백업 전략 가이드 수립.

결과

  • 100k 기준 업데이트 속도 최대 45% 성능 향상
  • 처리량(TPS) 약 80% 향상 — Pod 비율 최적화 및 메시지 분산 처리 구조 개선 효과

  • 통합 관측 체계 구축으로 장애 대응·운영 가시성 확보
